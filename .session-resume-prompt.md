### Workflow
- master から branch 必須、commit 末尾に `Co-Authored-By: Claude Opus 4.7 <noreply@anthropic.com>` + `Claude-Session: <session-url>` トレーラ
- PR は `gh pr merge N --merge --delete-branch` → master 同期(逐次自動マージ承認済)
- 新機能 / 大改修は brainstorming(`superpowers:brainstorming`)→ spec(`docs/superpowers/specs/`)→ writing-plans(`docs/superpowers/plans/`)→ TDD(inline executing-plans、単一モジュール密結合タスクは inline 推奨)
- **既存コード/データは所与でなく 4 軸で見直し改善提案**(memory `feedback_propose_improvements_to_existing`):brainstorming/設計時に発見した改善は明示提案 → ユーザー判断 → 採用判断(本 PR で / 別 PR / TODO)
- **PR/Merge 前必須**(`feedback_pr_merge_dqr_required` + CONTRIBUTING-modules.md「PR 検証ガイド」):
  1. **PR の性質を判定** → refactor PR / 新機能 PR / pure docs / 混合
  2. **適切な検証手段を選ぶ**:
     - refactor → byte-diff(US/JP p=2000 vs master、全 11 NDJSON sha256 IDENTICAL)
     - 新機能 → 3-axis DQR(US p≥10000 + JP p≥5000、構造 / 臨床整合 / JP 言語)
     - pure docs → regression(704 緑維持) + manual link review
  3. 必要修正実装
  4. **全 docs(EN README / JP README / DESIGN / 関連 module README / CLAUDE / TODO)を同時更新**(後追い docs PR を避ける = PR #79 教訓 / PR2 で確立)
  5. `docs/reviews/<date>-<topic>-data-quality-review.md` 作成(新機能 PR の場合)or `scratchpad/<topic>_byte_diff_results.md`(refactor PR の場合)
  6. PR 起票、body に audit / byte-diff サマリ + doc リンク

### 4 軸判断(`feedback_decision_axes`)
- データ品質 / 臨床整合 / メンテ性(責任分解点クリア)/ コンセプト適切性
- 「BNP-pattern surgical(formula only)」はリアリズム調整に限定。**設計バグの構造修正は別カテゴリ**(PR #62 / #74 / #78 / #81 / #82 / #83 / #84 が好例)
