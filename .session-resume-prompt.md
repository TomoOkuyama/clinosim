## Communication

- 日本語 default。「進めて」「おすすめで」= 継続execution信頼委譲
- Scope discipline(★★★): 明示的に「スコープを大きくせず」と言われた場合は、6軸評価でも
  narrow scope + データ品質/臨床整合性向上を軸にした候補のみ提示すること(session 35で
  確立)
- 修正のprocess weightは不確実性・リスクの大きさに合わせる(session 35で確立、上記
  「プロセス選択の教訓」参照)— 既存パターンの単純適用に毎回フルchainを回す必要はない
- Chain workflow(確立済、2バリエーション運用中):
  - **5-lens adversarial review方式**: recon → spec → TDD実装 → 独立検証 → PR →
    compact adv review(1-agent 5-lens、実証要求)→ fix → 全suite → merge
  - **subagent-driven-development方式**: recon → spec → plan(writing-plans、
    precedent file/canonical helper/ADRを明示調査した上でtask化)→ 4 task毎に実装
    subagent+reviewer subagentの2段ゲート → 最終whole-branch review(opus)→
    finishing-a-development-branch → merge
  - **直接軽量TDD方式**(session 35で新規確立): 既存パターンの単純適用かつ新規設計
    判断なしの場合、brainstorming/plan/subagent chainを省略してその場でTDD実装
    (RED→実装→GREEN→全suite実行→commit→push確認)。真の設計分岐点(例: CSV列名)が
    あれば、その一点だけユーザーに確認してから進める
