## ★★★ 自分自身のbackground検証コマンドに関する必須事項(session 37で新規確立)

**subagentでなく自分自身がBashで`run_in_background`実行したintegration/e2eテストが
完了する前に、対象ファイルの編集を続けないこと。** subprocessでCLIを起動するテスト
(`subprocess.run(["python", "-m", "clinosim.simulator.cli", "generate", ...])`)は
呼び出された瞬間のディスク上の内容を読むため、編集途中のファイル状態を掴むと無関係な
広範囲の失敗(session 37ではimaging/document系45件)が出て「巨大な回帰」に見える。
`run_in_background`でintegration系コマンドを起動したら、完了通知が来るまで対象ファイルは
編集せず、待つ間は読み取り専用の調査に充てること。疑わしい大量失敗が出たら、まず
「編集中に検証コマンドが動いていなかったか」を疑い、`git stash`でクリーンな状態と
比較して汚染かどうかを即座に切り分ける。詳細: memory
`feedback_own_background_verification_contamination.md`。

## Communication

- 日本語 default。「進めて」「おすすめで」= 継続execution信頼委譲
- Scope discipline(★★★): 明示的に「スコープを大きくせず」と言われた場合は、6軸評価でも
  narrow scope + データ品質/臨床整合性向上を軸にした候補のみ提示すること
- 修正のprocess weightは不確実性・リスクの大きさに合わせる — 既存パターンの単純適用に
  毎回フルchainを回す必要はない
- Chain workflow(確立済、2バリエーション運用中):
  - **5-lens adversarial review方式**: recon → spec → TDD実装 → 独立検証 → PR →
    compact adv review(1-agent 5-lens、実証要求)→ fix → 全suite → merge
  - **subagent-driven-development方式**: recon → spec → plan(writing-plans、
    precedent file/canonical helper/ADRを明示調査した上でtask化、self-reviewで
    sequencingバグも確認)→ 4 task毎に実装subagent+reviewer subagentの2段ゲート
    (subagent継続はSendMessage、成果物ファイルの実在はcontroller側でfind確認)→
    最終whole-branch review(opus)→ finishing-a-development-branch → PR → merge
  - **直接軽量TDD方式**: 既存パターンの単純適用かつ新規設計判断なしの場合、
    brainstorming/plan/subagent chainを省略してその場でTDD実装(RED→実装→GREEN→
    全suite実行→commit→push確認)。真の設計分岐点があれば、その一点だけユーザーに
    確認してから進める。session 37全体(データ品質6件+backlog16項目)がこの方式で
    完遂した実例 — 大量のタスクでも、個々が既存パターンの単純適用であれば
    subagent-driven-developmentの重量プロセスを経由せず直接実行できる
