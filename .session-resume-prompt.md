## ★★★ 作業方針(user 明示、session 40 wrap-up、session 41 以降デフォルト)

**CIF と FHIR の品質・整合性チェックとデバッグ、既存コードの修正を中心** に据える。
**新規追加は「FHIR 整合性の矛盾や異常を、新規追加をしなければ解消できない場合」に
限定** — 追加以外の解決策(既存コード修正 / データ triage / 既存 helper 経由化 /
検証 test 拡張 / YAML 修正等)で解決できるなら必ずそれを優先。「解決できる可能性が
ある」レベルで既存修正を試みるのではなく、**FHIR 整合性の矛盾を本当に既存修正で
解けるか探索した上で「不可能」と判断した時のみ**新規追加を提案する。

- **候補提示時の優先順**: 1) CIF/FHIR 監査 + 発見バグの **既存コード修正**、
  2) 既存 test / registry の横展開 sweep(既存 pattern 適用)、
  3) データ triage(delete + NOTE で silent-drop を晒す)、
  4) **既存修正で解決不可能と判断された FHIR 整合性矛盾への新規追加**(代替手法を
     具体的に列挙して否定した上で報告)、5) その他は明示要求時のみ。
- **A 群(daily_trajectory rich化 / β-2 手術記録 / cohort audit brainstorming 等)は
  デフォルトで着手しない**。ユーザーから明示切替指示があった場合のみ。
- **監査 primary 手法**: `python -m clinosim.simulator.cli generate --population <N>
  --country US/JP --seed 42 --output <path> --format fhir-r4` → NDJSON を FHIR validator /
  独自 grep(`(display unavailable)` / `表示不可` / 空 reference / 不整合 status /
  spec 違反 datetime 等)で監査。
- 「新規追加でしか解決できない」典型例(session 40 実績):
  - WHO ICD-10 I64 未登録 → codes/data/ 追加が唯一の解
  - CIF field 未 emit(no-drop 不変則違反)→ builder 追加が唯一の解
- 「新規追加は不要」で解けた典型例(session 40 実績、全 6 chain):
  - display fallback → 既存 `_map_diagnosis_code` 経由化(FP-FH-CODE-RESOLUTION)
  - silent-drop delta → 既存 canonical set に validator wire(FP-DELTA-VALIDATE)
  - inline duplication → 既存 helper 集約(FP-UNIFY-2/3)
- 詳細: memory `feedback_cif_fhir_quality_focus.md`(★★★ session 41 以降必読)

## Communication

- 日本語 default。「進めて」「おすすめで」= 継続execution信頼委譲
- Scope discipline(★★★): 明示的に「スコープを大きくせず」と言われた場合は、6軸評価でも
  narrow scope + データ品質/臨床整合性向上を軸にした候補のみ提示すること
- 修正のprocess weightは不確実性・リスクの大きさに合わせる — 既存パターンの単純適用に
  毎回フルchainを回す必要はない
- Chain workflow(確立済、3バリエーション運用中):
  - **5-lens adversarial review方式**: recon → spec → TDD実装 → 独立検証 → PR →
    compact adv review(1-agent 5-lens、実証要求)→ fix → 全suite → merge
  - **subagent-driven-development方式**: recon → spec → plan(writing-plans、
    precedent file/canonical helper/ADRを明示調査した上でtask化、self-reviewで
    sequencingバグも確認)→ 4 task毎に実装subagent+reviewer subagentの2段ゲート
    (subagent継続はSendMessage、成果物ファイルの実在はcontroller側でfind確認)→
    最終whole-branch review(opus)→ finishing-a-development-branch → PR → merge
  - **直接軽量TDD方式**: 既存パターンの単純適用かつ新規設計判断なしの場合、
    brainstorming/plan/subagent chainを省略してその場でTDD実装(RED→実装→GREEN→
    全suite実行→commit→push確認)。真の設計分岐点があれば、その一点だけユーザーに
