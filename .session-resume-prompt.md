## ★★★ subagentのbackground待機・継続に関する必須事項(session 36で追加確立)

- **同一subagentを継続する時は必ずSendMessage(to: <agentId>)を使う**。Agent toolを
  同じdescriptionで再度呼ぶと、前回の記憶を持たない別の新規subagentがspawnされるだけで、
  backgroundプロセス(pytest等)には接続されない。Agent tool結果に付与される
  `agentId: xxxxx (use SendMessage with to: 'xxxxx' to continue this agent)` の案内に
  必ず従うこと。
- controllerとsubagentは同じworktreeファイルシステムを共有している(別サンドボックス
  ではない)。subagentのbackground待機ping-pongが非効率になってきたら、**controller
  自身が直接Bashで`git log`/`git status`/`pytest`等を確認・実行して進行させる方が速い**
  ことが多い。ただし重複した重いコマンドを並行起動しないこと(host負荷が悪化する。
  重複に気づいたら`TaskStop`で自分の方を止める)。
- subagentが「DONE、report file作成済み」と自己申告しても、**controller側で`find`/`ls`
  により実際にファイルが存在するか確認する**こと。session 36ではtask-4-report.mdが
  実際には書かれていないまま「作成した」という報告が最終レビュアーの指摘で発覚した。
- 詳細: memory `feedback_subagent_worktree_and_background_task_pitfalls.md`(session 36

## Communication

- 日本語 default。「進めて」「おすすめで」= 継続execution信頼委譲
- Scope discipline(★★★): 明示的に「スコープを大きくせず」と言われた場合は、6軸評価でも
  narrow scope + データ品質/臨床整合性向上を軸にした候補のみ提示すること
- 修正のprocess weightは不確実性・リスクの大きさに合わせる(session 35で確立)—
  既存パターンの単純適用に毎回フルchainを回す必要はない
- Chain workflow(確立済、2バリエーション運用中):
  - **5-lens adversarial review方式**: recon → spec → TDD実装 → 独立検証 → PR →
    compact adv review(1-agent 5-lens、実証要求)→ fix → 全suite → merge
  - **subagent-driven-development方式**: recon → spec → plan(writing-plans、
    precedent file/canonical helper/ADRを明示調査した上でtask化、self-reviewで
    sequencingバグも確認)→ 4 task毎に実装subagent+reviewer subagentの2段ゲート
    (subagent継続はSendMessage、成果物ファイルの実在はcontroller側でfind確認)→
    最終whole-branch review(opus)→ finishing-a-development-branch → PR → merge
  - **直接軽量TDD方式**: 既存パターンの単純適用かつ新規設計判断なしの場合、
    brainstorming/plan/subagent chainを省略してその場でTDD実装(RED→実装→GREEN→
    全suite実行→commit→push確認)。真の設計分岐点があれば、その一点だけユーザーに
    確認してから進める
