## 1. プロジェクト概要

### 目的

病院における医療データのシミュレーション生成システムの構築。患者プロファイルの生成から、疾患イベント発生、入院中の診療プロセス、投薬・治療反応までをリアリティある形で再現する。

### リアリティの3次元

| 次元 | 内容 |
|---|---|
| 生物学的リアリティ | 人体の生理・病態として正しいか |
| 行動的リアリティ | 患者・医師の意思決定として自然か |
| システム的リアリティ | 医療制度・組織の中で起こりうるか |

## 2. 対象国・医療制度モデル

### 対象国

**日本・米国の2カ国**

### 制度パラメータ比較

| パラメータ | 日本 | 米国 |
|---|---|---|
| 保険制度 | 国民皆保険 | 民間保険中心（HMO/PPO） |
| ゲートキーパー | なし（専門医直接受診可） | 保険プランによる |
| 受診閾値 | 低い（軽症でも受診） | 高い（コスト意識が強い） |
| 大病院志向 | あり | なし（ネットワーク制限） |
| 重複受診 | 発生する | 保険制約で抑制 |
| 検査頻度係数 | 1.3（高め） | 0.8（低め） |
| 退院基準 | 検査値正常化まで待つ | 経口内服可能・解熱で早期退院 |
| 退院後管理 | 外来通院（月1〜2回） | 専門クリニック・遠隔モニタリング |
| 診療報酬制度 | DPC（診断群分類包括払い） | DRG（診断群別定額払い） |

### 患者行動モデル（国別）

#### 日本

- コンビニ受診（夜間・休日の軽症救急受診）
- 紹介状なしでも大病院に行く傾向
- 「念のため検査してほしい」要求が多い
- 家族の同意が治療方針に強く影響
- 社会的入院（医療的には退院可能だが帰る場所がない）

#### 米国

- 自己負担額（Deductible）を意識して受診を我慢
- ERは最終手段（コストが高い）
- Urgent Careを中間的選択肢として活用
- 処方薬コストを理由に服薬アドヒアランスが低下
- 保険会社による入院事前審査（Prior Authorization）が存在
- 防衛的医療（訴訟リスクを意識した過剰検査）

### DPC/DRGによる在院日数圧力

#### 日本（DPC）

```
入院期間Ⅰ（短い）：高い包括点数
入院期間Ⅱ（標準）：標準包括点数  ← 病院はここまでに退院させたい
入院期間Ⅲ（長い）：低い包括点数
```

#### 米国（DRG）

```
在院日数に関係なく疾患で定額報酬
→ 短いほど病院の利益になる
→ 30日再入院はペナルティ（ACA規定）
→ 退院後ケアマネジメントが発達
```

### 転院・ケア移行モデル

#### 日本の流れ

```
急性期病院（DPC病院）
  ↓ 病状安定後
回復期リハビリ病院（脳卒中・骨折）
または療養型病院（長期管理）
または在宅（訪問診療・訪問看護）
または老人保健施設
```

#### 米国の流れ

```
急性期病院
  ↓ 早期退院
Skilled Nursing Facility（SNF）← 短期リハビリ
または Inpatient Rehabilitation Facility（IRF）
またはHome Health Agency（訪問看護）
または Hospice（終末期）
```

## 3. 患者プロファイル設計

### 人口統計学的属性

- 年齢・性別分布：実際の人口統計に準拠（高齢者ほど多病）
- 家族歴・遺伝的リスク因子の継承（例：糖尿病の家族歴）
- 社会経済的背景（保険種別、居住地域）
- 就労状況（入院期間の短縮圧力に影響）
- 独居か家族同居か（退院後管理に影響）
- 交通アクセス（外来フォロー受診率に影響）

### ベースライン健康状態

- 既往歴・慢性疾患の組み合わせはリアルな共起パターンで生成
  - 例：高血圧＋糖尿病＋脂質異常症は高頻度で共存
- バイタルサインのベースライン値は年齢・性別・BMI・疾患と相関
- アレルギー・禁忌薬剤の設定

### 生理的個人差パラメータ（`PatientPhysiologicalProfile`）

患者生成時に一度だけ確率的に決定される「隠れた体質」。その後の全ての反応を規定する。

| パラメータ | 内容 | 分布 |
|---|---|---|
| `immune_reactivity` | 免疫応答の強さ（炎症反応の個人差） | Beta(5,5) |
| `drug_metabolism_rate` | 薬物代謝速度（CYP酵素多型） | poor 7% / normal 70% / rapid 15% / ultra_rapid 8% |
| `renal_reserve` | 腎臓の予備能 | Beta(8,2)＋年齢補正 |
| `cardiac_reserve` | 心臓の予備能 | Beta(8,2)＋年齢補正 |
| `hepatic_reserve` | 肝臓の予備能 | Beta(8,2)＋年齢補正 |
| `treatment_sensitivity` | 薬剤感受性（抗生剤・利尿剤・鎮痛剤） | Normal(1.0, σ) |
| `symptom_reporting_bias` | 症状表現傾向（過大/過小報告） | Normal(1.0, 0.3) |
| `delirium_susceptibility` | せん妄リスク | Beta(2,8) |
| `dvt_susceptibility` | 深部静脈血栓リスク | Beta(2,8) |

### Health Literacy（健康リテラシー）

- 高い → 症状を正確に伝える、服薬を守る
- 低い → 症状の過小/過大申告、指示を理解しない

## 4. イベント発生モデル

### 時間的リアリティ

- 季節性を反映（インフルエンザ→冬、熱中症→夏）
- 年齢別疾患発症率（ハザード関数による時系列）
- 慢性疾患の急性増悪はランダムサージで発生

### 疾患の重症度分類と状態遷移

```
軽症（外来のみ）→ 中等症（入院）→ 重症（ICU）
```

### 合併症の連鎖モデル

- 例：肺炎 → 敗血症 → 多臓器不全
- 在院日数が長いほど発生率が上がる二次イベント

| 二次イベント | 主なリスク因子 |
|---|---|
| 院内感染（MRSA、CDiff） | 在院5日目以降 |
| 深部静脈血栓（DVT） | 臥床日数 |
| 褥瘡 | Braden スコア |
| せん妄 | 高齢・ICU・睡眠障害 |
| 誤嚥性肺炎 | 嚥下機能低下 |

### 受診行動モデル

```
症状発生
  ↓
症状の認知（閾値：個人差大）
  ↓
自己対処（市販薬、様子見）
  ↓
受診判断トリガー：
  - 症状が3日以上続く
  - 日常生活に支障が出る
  - 発熱38.5℃超
  - 家族・周囲に促される
  ↓
受診先の選択（国・保険・時間帯・距離で分岐）
```

### 再入院モデル

30日再入院率の現実値をシミュレーションが再現できることを品質指標とする。

| 疾患 | 30日再入院率 |
|---|---|
| 心不全 | 約20〜25% |
| 肺炎 | 約15〜20% |
| COPD | 約20% |

## 6. 検査値生成エンジン

### 基本設計思想

```
NG：各検査値を独立にランダム生成
    → 臨床的に矛盾した値が生成される

OK：「生理的状態変数」を先に定義し、そこから検査値を導出
    → 検査値間の相関が自然に保たれる
```

### 3層アーキテクチャ

```
Layer 1：生理的状態空間
（隠れ変数：直接観測されない）
        ↓
Layer 2：検査値導出エンジン
（状態変数 → 検査値の変換ロジック）
        ↓
Layer 3：観測ノイズ・欠損・タイミングズレ
（測定誤差・欠損・時刻のずれ）
```

### Layer 1：生理的状態変数

| 変数名 | 範囲 | 説明 |
|---|---|---|
| `inflammation_level` | 0.0〜1.0 | 0=正常、1=重篤な炎症 |
| `renal_function` | 0.0〜1.0 | 1=正常(eGFR>90)、0=無尿 |
| `cardiac_function` | 0.0〜1.0 | 1=EF>60%、0=心原性ショック |
| `hepatic_function` | 0.0〜1.0 | 肝機能スコア |
| `anemia_level` | 0.0〜1.0 | 0=正常、1=重篤な貧血 |
| `coagulation_status` | 0.0〜1.0 | 0=正常、1=DIC |
| `volume_status` | -1.0〜+1.0 | -1=重篤な脱水、+1=著明な溢水 |
| `perfusion_status` | 0.0〜1.0 | 1=正常、0=ショック |
| `ph_status` | -1.0〜+1.0 | -1=重篤なアシドーシス、+1=アルカローシス |

### Layer 2：検査値の相関構造（主要な依存関係）

#### 炎症系の時間的遅延

```
感染 → WBC上昇（6〜12時間）
     → CRP上昇（24〜48時間遅延）
     → プロカルシトニン上昇（3〜6時間、細菌感染特異的）
     → ESR上昇（48時間以降、正常化も最遅）
     → アルブミン低下（数日単位）
```

#### 腎機能の連鎖

```
Cr上昇 → BUN上昇（BUN/Cr比 10:1〜20:1）
       → eGFR低下（CKD-EPI計算式）
       → K上昇（排泄障害）
       → HCO3低下（代謝性アシドーシス）
       → Ca低下（活性型VitD産生低下）
       → P上昇（蓄積）
```

#### バイタルサインの複合的連動

```
体温 = ベースライン + 炎症レベル×3.0 + 日内変動（正弦波）
心拍数 = ベース + (体温-37.0)×10 + (1-perfusion)×40
血圧 = ベース + volume_status×15 + perfusion_effect
```

### Layer 3：観測ノイズと欠損パターン

#### 測定誤差モデル

- 正規分布ノイズ（変動係数CVは検査項目ごとに設定）
- 2%の確率で外れ値（採血ミス・溶血・機器エラー）

#### 欠損パターン（ランダムではなく文脈依存）

| 欠損要因 | 欠損確率 |
|---|---|
| 夜間・深夜（22時〜6時） | 緊急以外の検査は80%欠損 |
| 患者が検査拒否 | 30%欠損 |
| 採血困難（高齢者・肥満） | 10%失敗→再採取 |
| 抗生剤投与後の培養採取 | 感度が60%に低下（偽陰性） |

#### タイミングのズレ

```
オーダー時刻 → 採血時刻（+30〜60分）
             → 結果報告時刻（+1〜2時間）
             → 医師確認時刻（+さらに遅延）

血液培養：採取から72時間後に最終結果
画像読影：緊急30分、通常2〜4時間
病理：3〜7日
```

#### 「説明のつく異常値」パターン

| パターン | 影響する検査値 | 手がかり |
|---|---|---|
| 採血時の溶血 | K偽高値、LDH上昇 | 検体備考「溶血」 |
| 点滴ライン近傍採血 | 電解質希釈 | 再採血で正常化 |
| 食後採血（2時間以内） | 血糖+30〜60、TG著明上昇 | 採取時刻との照合 |
| Biotin（高用量）服用 | 甲状腺ホルモン・トロポニン干渉 | 薬歴との照合 |

### 臨床的経過アーキタイプ

| アーキタイプ | 確率 | 内容 |
|---|---|---|
| `smooth_recovery` | 55% | 順調に改善 |
| `dip_then_recovery` | 20% | 一時悪化後に改善（Day2頃に谷） |
| `plateau_then_recovery` | 10% | 横ばい後に改善 |
| `treatment_resistant` | 8% | 初期治療に反応せず変更が必要 |
| `gradual_deterioration` | 5% | 緩徐に悪化 → ICU移送 |
| `sudden_deterioration` | 2% | 突然の重篤化 |

### 未知の変化に対する生成戦略（優先順位）

```
Priority 1：既知の病態生理モデルで説明できるか
  → YES：Layer 1〜2のルールで生成

Priority 2：個人差パラメータで説明できるか
  → YES：PhysiologicalProfileのパラメータで調整

Priority 3：既知の臨床パターン（アーキタイプ）に当てはまるか
  → YES：アーキタイプから選択

Priority 4：どれにも当てはまらない純粋な「未知」
  → 文脈制約（変化速度上限・相互排他条件）の範囲内で
    制約付きランダム生成
    + 「説明のつく異常値」パターンを付与
```

### 生理的制約（バリデーションルール）

#### 変化速度の上限（1時間あたり）

| マーカー | 上昇上限 | 低下上限 |
|---|---|---|
| CRP | 20 mg/L/h | 5 mg/L/h |
| Creatinine | 0.1 mg/dL/h | 0.02 mg/dL/h |
| 体温 | 0.3 ℃/h | 0.2 ℃/h |
| Hemoglobin（輸血なし） | 0.05 g/dL/h | 0.3 g/dL/h |

#### 相互排他条件（同時に起こりえない組み合わせ）

- WBC > 15000 かつ CRP < 5 → 矛盾
- pH < 7.2 かつ 呼吸数 < 16 → 矛盾
- ショック状態（perfusion < 0.3）かつ SBP > 110 → 矛盾

#### パニック値（緊急報告値）

| 検査値 | 閾値 | 対応アクション |
|---|---|---|
| K | > 6.5 mEq/L | 即座に医師へ電話 |
| Hb | < 7.0 g/dL | 輸血検討オーダー |
| 血糖 | > 500 mg/dL | 緊急インスリン投与 |

## 7. 診断的推論エンジン

### 設計思想

現実の医療は「症状 → 診断確定 → 治療」ではなく、トライアンドエラーによる確率的な診断プロセスである。これをベイズ的な確率更新でモデル化する。

```
症状 → 鑑別診断リスト（確率分布）
     → 検査 → 確率更新
     → 追加検査・治療的診断
     → 確率が閾値超 → 診断確定
     → または確定できないまま退院
```

### 3フェーズの診断プロセス

#### Phase 1：仮説生成（鑑別診断リスト）

- 主訴から候補疾患を列挙
- 各候補の事前確率（Prior）を設定
  - 年齢・性別・季節・既往歴・曝露歴で調整

#### Phase 2：仮説検証（ベイズ的更新）

検査結果が出るたびに各疾患の確率を更新する。

```
P(Disease|Test) ∝ P(Test|Disease) × P(Disease)
           ↑尤度比（LR）      ↑事前確率
```

#### Phase 3：診断収束

確率が閾値を超えた時点で診断確定（または未確定のまま治療継続）。

### 治療的診断（Therapeutic Trial）

| 試験 | 適応 | 判定時間 | 陽性の意味 |
|---|---|---|---|
| 抗生剤トライアル | 肺炎疑いだが起因菌不明 | 72時間 | 細菌性感染の可能性が高い |
| 利尿剤トライアル | 心不全vs肺炎の鑑別困難 | 4〜6時間 | 心原性肺水腫が主体 |
| ステロイドトライアル | 原因不明の炎症・発熱 | 48〜72時間 | 自己免疫・炎症性疾患を示唆 |

### 診断の「迷走」パターン

| パターン | 確率 | 例 |
|---|---|---|
| 初期誤診 → 修正 | 15% | 肺炎 → 肺癌+閉塞性肺炎（CTで腫瘤発見） |
| 複数疾患の同時存在 | 20% | 肺炎 + 心不全増悪（高齢者に多い） |
| 診断未確定のまま退院 | 10% | 発熱の原因精査中→外来で継続 |
| 偶発的発見 | 8% | 肺炎入院中のCTで腎腫瘤 |

### 診断プロセスがデータに与える影響

- 検査の「順序と意図」が整合する（Day0：広くスクリーニング → Day3：確認検査）
- 治療変更に文脈がある（「3日間改善なし → 抗生剤変更」の根拠がデータに残る）
- 診断名が時間とともに変化する（「肺炎疑い」→「肺炎（細菌性）」→「肺炎（肺炎球菌性）」）
- ナラティブな記録が検査結果と連動する

## 8. 投薬モデル

- 疾患ガイドライン準拠の第一選択薬・代替薬ロジック
- 用量は体重・腎機能・年齢で自動調整
- アレルギー・禁忌チェック
- 薬物相互作用の考慮
- 治療反応の確率的モデル化
  - 効果発現タイミング（例：抗生剤は72時間後に判定）
  - 治療失敗率・副作用発現率
  - 治療変更トリガー

### 服薬アドヒアランスモデル

| 状況 | アドヒアランス率 |
|---|---|
| 入院中 | ほぼ100%（看護師管理） |
| 退院後（標準） | 75〜85% |
| 退院後（米国・高コスト薬） | 50〜65% |
| 退院後（高齢者・多剤） | 60〜75% |

## 9. データ標準規格

### コード体系一覧

| 項目 | 日本 | 米国・国際 | clinosim実装 |
|---|---|---|---|
| 病名コード | ICD-10 (WHO) | ICD-10-CM | `clinosim/codes/data/icd-10-cm.yaml`, `icd-10.yaml` |
| 薬剤コード | YJコード | RxNorm | `clinosim/codes/data/yj.yaml`, `rxnorm.yaml` |
| 検査コード | JLAC10 | LOINC | `clinosim/codes/data/jlac10.yaml`, `loinc.yaml` |
| 診療行為 | Kコード | CPTコード | `clinosim/codes/data/k-codes.yaml`, `cpt.yaml` |
| 単位 | UCUM | UCUM | (loader内蔵 system URI) |
| データ形式 | HL7 FHIR R4 Bulk Data NDJSON | HL7 FHIR R4 Bulk Data NDJSON | one .ndjson per resource type |

### コード体系モジュール (`clinosim/codes/`)

すべての臨床コードは `clinosim/codes/` モジュールが一元管理 (locale非依存)。 各コードは英語 (必須) + 日本語 (オプション) の多言語表示を持ちます。

```python
from clinosim.codes import lookup, get_system_uri

lookup("icd-10-cm", "N10", "en")  # → "Acute tubulo-interstitial nephritis"
lookup("icd-10-cm", "N10", "ja")  # → "急性腎盂腎炎"
get_system_uri("loinc")            # → "http://loinc.org"
```

### 権威ソース (Authoritative Sources)

| Code System | 出典 |
|---|---|
| ICD-10-CM | [CMS ICD-10-CM 2024](https://www.cms.gov/medicare/coding-billing/icd-10-codes) |
| ICD-10 (WHO) | [WHO ICD-10](https://icd.who.int/browse10/) |
| LOINC | [Regenstrief LOINC](https://loinc.org/) |
| JLAC10 | [日本臨床検査標準協議会 (JCCLS)](https://www.jccls.org/) |
| RxNorm | [NLM RxNorm](https://www.nlm.nih.gov/research/umls/rxnorm/) |
| YJ コード | 厚生労働省 薬価基準 |
| CPT | [AMA CPT](https://www.ama-assn.org/practice-management/cpt) |
| K コード | 厚生労働省 診療報酬点数表 |

### CIF と表示テキストの分離

CIF (Clinosim Intermediate Format) は **コード値のみ** を保持し、 表示テキストは出力時に codes module 経由で解決されます (AD-30):

```python
@dataclass
class ClinicalDiagnosis:
    admission_diagnosis_code: str = ""               # 例: "N10"
    admission_diagnosis_system: str = "icd-10-cm"    # コード体系キー
    discharge_diagnosis_code: str = ""
    discharge_diagnosis_system: str = "icd-10-cm"
    # display は保持しない (CIF はコードのみ)
```

FHIR / CSV / narrative 出力時に各 adapter が `clinosim.codes.lookup()` を呼んで言語別の display を取得します。

## 10.1 出力フォーマット (v0.1-beta)

### CIF (Clinosim Intermediate Format)

シミュレーションの **immutable intermediate format**。 1 encounter = 1 JSON ファイル。 すべての出力 adapter はここから派生。

```
output/cif/
├── metadata.json
├── hospital.json                  # staff roster + hospital config
└── structural/patients/
    └── ENC-POP-XXXXXX-NNNNNN.json
```

### FHIR R4 — Bulk Data Export NDJSON 形式

[HL7 FHIR Bulk Data Access](https://hl7.org/fhir/uv/bulkdata/) 仕様準拠。 1 リソース型 = 1 NDJSON ファイル。

```
output/fhir_r4/
├── manifest.json                   # Bulk Data manifest
├── _facility.json                  # Organization + Location マスター
├── Patient.ndjson                  # 1 患者 1 行
├── Encounter.ndjson
├── Observation.ndjson              # labs + vitals (LOINC)
├── Condition.ndjson                # 主疾患 + 慢性疾患 (ICD-10-CM)
├── MedicationRequest.ndjson        # RxNorm
├── MedicationAdministration.ndjson
├── Procedure.ndjson                # CPT
├── AllergyIntolerance.ndjson
├── Practitioner.ndjson
├── PractitionerRole.ndjson         # specialty + ward location
├── Organization.ndjson             # hospital + departments
└── Location.ndjson                 # wards (wa) + beds (bd)
```

各 Resource.id は 12 リソース型すべてで一意。 reference 整合性が取れています。

### CSV

- patients, encounters, conditions, lab_results, vital_signs, orders, medication_administrations, procedures, ...

## 10.2 Snapshot date semantics (v0.1-beta)

`--end YYYY-MM-DD` 引数で **snapshot date** を指定可能 (default: 今日)。

- snapshot date を超える life event は生成されない
- snapshot 時点で discharge_datetime が未来になる入院:
  - `Encounter.status = "in-progress"`
  - `discharge_datetime = None`
  - 部分的な lab/vital/order/MAR データのみ
  - 主疾患 `Condition.clinicalStatus = "active"` (resolved されない)
- これにより**現在入院中の患者**を含む現実的な EHR スナップショットを生成可能 (NEWS2 アラートシステムなどに対応)

## 10.3 病院構成 (Hospital Configuration)

`clinosim/config/hospital_*.yaml` で病院の物理レイアウトを定義:

| 設定項目 | 内容 |
|---|---|
| `recommended_population` | 推奨 catchment 人口 |
| `available_departments` | 利用可能な診療科 (内科、循環器、消化器、外科、整形、救急、プライマリケア) |
| `department_rollup` | 専門科 → 利用可能科への集約マップ (e.g., pulmonology → internal_medicine) |
| `wards` | 各科の病棟 ID リスト |
| `ward_capacity` | 病棟ごとのベッド数 |
| `resource_capacity` | lab analyzer / CT / MRI / OR 等の数 |
| `staffing` | シフト別 staffing 比率 |

これにより:
- 疾患 → 科 → 病棟 → ベッドの自動 routing
- M/M/1 queueing で検査結果遅延が動的に変動
- 病棟ごとに看護師配属 (FHIR PractitionerRole.location)
- 標準セット (50床、 10床) を切り替え可能
